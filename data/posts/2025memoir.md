## 0 to 1

### FSD 구조 확립

7월 16일 입사당시 회사는 큰 그림만 그려진 상황이었다.

프론트엔드에서는 초기 디자인과 로그인 페이지만 되어있었고, 구조도 잡히지 않아 페이지나 컴포넌트를 만들수록 추후 유지보수에 비용이 더 커질것이라 판단했다.

그래서 다양한 팀 프로젝트를 하며 확장성있는 아키텍처에 고민하여 FSD 구조를 써왔고, 해당 아키텍처의 장점을 대표님께 어필하여 아키텍처를 FSD로 전면 교체하게 되었다.

이 선택은 정말 좋은 결과를 불러왔다. FSD 구조에 대한 설명과 이해가 필요했지만 알고나면 적용하기 편했다.

그렇게 몇달정도 구조를 쌓아오는 과정에서 순환참조 문제가 발생했다. 근원지를 찾아보니 FSD 구조지만, 정작 룰이 지켜지지 않는 부분들이 꽤 있었다.

FSD-guard가 도입되고, 공식문서에 맞춰 룰을 만들었는데 **최소 150개 이상의 컴포넌트**가 룰을 어기고 있었다. FSD의 기본적인 rule인 하위 컴포넌트에서 상위 컴포넌트를 import하면 안되는 룰도 **34개**나 되었다.

이 때 깨달은 점은 개발자가 인지만 한다고 모든 것을 컨트롤할 수 없고, 룰이나 제도를 만들어 관리를 해야 되는 것을 느꼈다.

주요 기능을 먼저 개발 다 한뒤, guard에 위반했던 컴포넌트를 **1주일 내로 위반 컴포넌트 0개**를 만들어 깔끔한 구조를 형성할 수 있었다. 구조를 통해 순환적 참조를 해결할 수 있었다.

### API Script 생성

api를 개발할 떄 AI에 의존하여 rules_mdc와 api 컨텍스트를 주어 생성하였다. 하지만 때때로 다른 형식의 api를 생성했고 token 역시 낭비되는 경우가 많았다. 이는 추후 팀원이 추가될수록 merge_conflict 생길 위험과 api 생성 비용이 크다고 판단하여 script를 만들어보기로 했다.

서버에서 생성한 endpoint 및 request, response 그리고 모든 type들이 담겨있는 yml 파일이 있었다. 파일 구조를 파악한 후 dfs 방식을 써서 response및 request type을 정의하고, api에서 model을 import하여 정의된 타입으로 api를 만드는 스크립트를 만들었다.

생성된 API는 FSD 구조의 `entities` layer 내 각 `domain` slice 폴더의 `api` segment에 배치되었다. api에 필요한 type들은 `model` segment를 만들어 정의했다.

- `model / index.ts`: api에 필요한 모든 type 파일
- `api / queryKey.ts`: 쿼리키 정리 파일
- `api / relations.ts`: mutation 시 연관된 query invalidate를 위한 파일
- `api / index.ts`: api 모음 파일

중간중간 script 수정이 이뤄졌지만 결과적으로 사람이 api를 만들지 않아 conflict 생길 염려가 없었고, 서버에서 **100개의 api**를 만든다고 해도 **10초도 안되서** 모든 api와 type들이 만들어졌기 때문에 개발 속도가 측정이 안될 정도로 굉장히 빨라졌다. 이는 팀원이 늘어날수록 더 큰 가치를 만들수 있다고 생각한다.

### cursor 및 AI 활용도

cursor를 활용해 코드 작성의 효율성을 높였다. cursor를 통한 vibe coding에서 중요한 건 프롬프트 작성인 것 같다.

먼저 하나의 샘플을 잘 가다듬고 그 형식에 맞춰 비슷한 기능 생성 및 수정을 했을 때, 통일성 있게 작성이 되었다. 예를 들면 login-form을 만들 때 zod로 타입 정의를 하고, react-hook-form으로 form validation 및 error 메시지 관리 등을 잘 만든 후, 다른 form을 만들 때 이를 참고하여 만들었을 때 완성도 있는 결과물이 나왔다.

cursor pro 2개 계정을 쓰면서 느낀점은 context 정리가 중요하다. 이를 알기 전 20일이 채 지나지 않아 모든 토큰을 다 사용하였다.

이때 cursor 토큰 사용량을 보니 질문이 10개가 연속적으로 넘어갈수록 기하급수적으로 토큰 사용량이 늘어났다. ai가 쓸데없는 정보를 많이 알고 있을 때 token이 많이 사용되므로 주제별로 context를 나누는 게 중요한 거 같다. 그래야 할루시네이션도 줄어들고, 쓸데없는 코드를 만들지 않는다.

가끔 이전 컨텍스트를 기억하여 수정을 하여 나도 모르게 커밋이 이뤄지는 경우가 있는데 정말 주의해야한다.

### 테스트

기능이 생길수록, 예상치 못한 버그들이 나왔다. 점점 리팩토링하는 과정에서도 안정성이 떨어졌고, GS 인증을 받기 위해 테스트가 필요했다.

테스트에 많은 시행착오를 겪었다. unit, integration, e2e 등과 같은 기능테스트와 chromatic, snapshot, storybook 등을 통한 시각적 테스트 등 많이 만들어보고 수정했었다.

결론적으로는 integration, e2e 기능테스트 위주로 만들었다. unit 테스트는 다른 컴포넌트와 연계가 없어 즉각적으로 보여지는 대로 테스트가 가능했기 때문에 제외하였다. 시각적 테스트는 design이 자주 바뀌고 아직 확정되지 않은 부분들이 많아 현재로선 비용만 증가한다고 판단했다.

integration 테스트는 hooks, lib, ui 등 기능에 따라 세밀하게 테스트를 만들었고, api mocking 라이브러리인 msw를 통해 서버로부터 어떤 값이 올 때 잘보여지는지, 기능이 잘 작동되는지 등을 작성했다.

## 1 to N을 향하여

코엑스 행사를 마무리 지은 후 그동안 미뤄왔던 기술부채들을 정리하고 내실을 다졌다.

saas 환경 뿐만 아니라 on-premise 환경도 설정하여 B2C, B2B 서비스를 시작하는 기반을 다지고 2026년부터 큰 도약을 하길 기다리고 있다.

감사하게도 대표님께서 지금까지 성과를 인정해주셔서 **2025년 11월 3일 입사한지 111일만에** 프론트 리드로 정식 임명받았다. 회사가 해주는 최고의 배려라고 생각한다.

그만큼 열심히 했고, 작성한 코드에 대한 확신도 든다. 이제는 누구에게 설명하더라도 현재 아키텍처에 대한 견고함에 대해 설명할 수 있을 만큼 코드에 대한 확신이 커지고 있다.

아직은 나 포함 하여 프론트 팀원은 **2명** 뿐이지만, 프론트엔드 리드로서의 역할에 대해 공부하고, 나의 퍼포먼스를 늘리는 것만큼이나 팀의 퍼포먼스를 늘리는 부분에 초점을 두어 어떻게 하면 더 성장할 수 있을지 고민하고 공부하고 있다.

## 2026년 목표

회사에서의 목표는 GS인증을 받고 많은 공공기관이나 기업에서 우리 제품을 사용하게끔 돕고 싶다.

개인적으로는 개발 공부를 지속적으로 하면서 프론트뿐만 아니라 여러 기술을 접하며 개발의 시야를 넓히고 싶다.
